# Agent 协作设计实践

---

## 背景

本文档记录使用 AI Agent 协作管理技术档案的实战经验，聚焦于**如何避免过度设计**和**保持文档一致性**。

---

## 核心原则

### 1. DRY（Don't Repeat Yourself）

**定义**：信息只在一处维护

**违反后果**：
- 多处修改不同步
- 读者看到矛盾的信息
- 维护成本指数级增长

**案例**：
```
❌ 错误：PROJECT.md 和 QUICKSTART.md 都包含导航链接
✓ 正确：只保留 QUICKSTART.md，删除 PROJECT.md
```

---

### 2. YAGNI（You Aren't Gonna Need It）

**定义**：不创建"可能有用"的文件

**违反后果**：
- 冗余文件堆积
- 初始复杂度增加
- 后续删除成本

**案例**：
```
❌ 错误：觉得"完整项目应该有 PROJECT.md"就创建了
✓ 正确：等用户说"需要一个入口文档"再创建
```

---

### 3. 自动化优先

**定义**：能自动的绝不手动

**实现方式**：
- Git Hook 被动触发
- 脚本集中管理
- 目录结构即数据

---

## 踩坑记录

### 坑 1：冗余文档

**现象**：
创建了 `PROJECT.md` 和 `QUICKSTART.md`，两者功能重叠。

**时间线**：
```
用户要求：脚本放到 .scripts 目录
AI 想到：还需要一个项目入口文档
AI 做了：创建 PROJECT.md
结果：与 QUICKSTART.md 重复
```

**根因分析**：

| 心理 | 表现 |
|------|------|
| 害怕遗漏 | "可能用户需要入口" |
| 过度补偿 | 给得比要求的多 |
| 完整性驱动 | "应该有个 PROJECT.md" |

**代价**：
- 创建：~30 秒
- 发现冗余：用户时间
- 删除 + 同步：~40 秒
- **总计**：~1 分钟 + 信任成本

**避免方法**：
1. 只做用户明确要求的
2. 认为需要额外的，先问
3. 宁可"少给"，不要"多给"

---

### 坑 2：文档一致性维护

**现象**：
删除 `PROJECT.md` 后，需要在其他文档中删除对它的引用。

**连锁反应**：
```
删除 PROJECT.md
  ↓
QUICKSTART.md 中有引用 → 需要修改
README.md 中有引用 → 需要修改
.agent/SKILL.md 中有引用 → 需要修改
```

**根因**：
- 文档之间有交叉引用
- 没有单一"真相源"

**解决方案**：
1. 减少文档数量
2. 交叉引用集中到一处
3. 使用脚本自动生成索引

## 文档分层经验

| 文档 | 目标读者 | 内容边界 | 更新方式 |
|------|----------|----------|----------|
| `QUICKSTART.md` | 人类 | 30 秒上手，常用命令 | 手动 |
| `.agent/SKILL.md` | Agent | 完整工作流程，扫描规则 | 手动 |
| `.scripts/README.md` | 开发者 | 脚本使用说明 | 手动 |
| `README.md` | 所有人 | 索引（自动生成） | 自动 |

**关键洞察**：
- 人类不需要看 Agent 的工作细节
- Agent 不需要看人类的快速指南
- **索引必须自动生成**，否则一定会过时

## AI 协作模式

### 有效模式

```
用户：明确需求
  ↓
AI：确认理解（如有歧义）
  ↓
AI：执行最小可用方案
  ↓
用户：反馈
  ↓
AI：迭代
```

### 无效模式

```
用户：需求 A
  ↓
AI：想到 A+B+C 都可能需要
  ↓
AI：做了 A+B+C
  ↓
用户：B 和 C 是冗余
  ↓
AI：删除 B+C
```

---

## 检查清单

创建新文件前，AI 应该自问：

- [ ] 这是用户明确要求的吗？
- [ ] 现有文件能实现同样功能吗？
- [ ] 如果我不问，用户会想到这个需求吗？
- [ ] 这个文件的维护成本是多少？
- [ ] 信息是否在其他地方已经存在？

## 总结

**核心教训**：

> "不要替用户决定什么是必要的"

**行动准则**：

1. 只做明确要求的
2. 需要额外的先问
3. 宁可少给不要多给
4. 自动化能解决的绝不手动
