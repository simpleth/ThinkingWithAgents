# OpenClaw 核心架构深度解析

---

## 概述

OpenClaw（原名 Moltbot、Clawdbot）是一个开源、自托管的 AI Agent 框架，由奥地利开发者 Peter Steinberger 创建。它将大语言模型转化为持久化、可使用工具的智能代理，并通过标准化接口连接到日常通讯应用（如 WhatsApp、Telegram、飞书、钉钉等）。

**为什么值得研究**：OpenClaw 代表了一种新兴的 AI 应用范式——将 AI 助手作为基础设施而非单纯的聊天机器人。其架构设计体现了 2026 年 AI 工程化的最佳实践：插件化、本地优先、记忆驱动、技能可扩展。

**核心结论预览**：
- 四层解耦架构（交互层、网关层、智能体层、执行层）确保高度可扩展性
- 三级记忆系统（短期日志 + 近端会话 + 长期知识）实现高效上下文管理
- 插件化 Provider 设计支持 22+ 模型动态加载
- 基于 SQLite + sqlite-vec 的混合检索兼顾性能与本地化

---

## 一、整体架构设计

### 1.1 四层架构模型

OpenClaw 采用经典的分层架构，从外到内依次为：

```
┌─────────────────────────────────────────────────────────────┐
│                    交互层 (Interaction)                      │
│   Telegram │ WhatsApp │ 飞书 │ 钉钉 │ Discord │ Slack ...   │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                      网关层 (Gateway)                        │
│         会话路由 │ 消息队列 │ 定时任务 │ 连接管理            │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                    智能体层 (Agent)                          │
│    ReAct 循环 │ 三级记忆 │ 意图理解 │ 任务拆解 │ 反思       │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                    执行层 (Execution)                        │
│      Skills │ Nodes │ Tools │ 沙箱隔离 │ 远端设备控制       │
└─────────────────────────────────────────────────────────────┘
```

**设计原则**：
- 每层职责清晰，通过标准化接口通信
- 支持单机运行，也支持分布式扩展
- 插件化设计，各层均可独立演进

### 1.2 技术栈概览

| 类别 | 技术选型 |
|------|----------|
| **核心语言** | TypeScript (全栈) |
| **运行时** | Node.js ≥ 22 |
| **Web 框架** | Express, Hono |
| **数据存储** | SQLite + FTS5 + sqlite-vec |
| **消息队列** | BullMQ (基于 Redis) |
| **浏览器自动化** | Playwright (语义快照) |
| **部署** | Docker (沙箱隔离) |
| **日志** | pino-pretty (开发), JSON (生产) |

## 二、交互层 (Interaction Layer)

### 2.1 职责

对接 25+ 通讯渠道，将各平台消息统一转换为内部标准格式，为上层提供一致的 API。

### 2.2 核心组件

**Channel 适配器**：
- 采用适配器模式，每个渠道实现统一接口
- 核心方法：`initialize()`、`sendMessage()`、`handleEvent()`
- 支持 WebSocket 长连接和 Webhook 两种模式

**连接管理器 (ConnectionManager)**：
- 动态管理各渠道连接状态
- 智能切换连接策略（WebSocket / 长轮询 / FRP 内网穿透）
- 心跳检测（30s 间隔）与断线自动重连

### 2.3 路由策略

| 策略 | 说明 |
|------|------|
| `dmPolicy` | 私聊直连策略：pairing / allowlist / open / disabled |
| `mentionGating` | 群聊 @触发机制，防止误响应 |

## 三、网关层 (Gateway Layer)

### 3.1 职责

作为系统的"交通警察"，负责会话路由、消息队列管理、定时任务调度。

### 3.2 会话管理

**Per-channel-peer 模式**：
- 同一用户在不同渠道拥有独立 Session
- 避免上下文混淆（如 WhatsApp 和 Telegram 的对话隔离）
- 基于 `channelId + sessionId` 映射 Agent 实例
- 路由表存储于 Redis

**SessionManager 核心功能**：
- Session 创建、更新、持久化
- 上下文压缩 (`compressContext()`)
- 状态同步与生命周期管理

### 3.3 消息队列

**车道式队列设计**（基于 BullMQ）：
- 会话内串行：保证同一对话的顺序性
- 会话间并行：提高整体吞吐量
- 支持延迟任务与死信重试

**并发控制**：
- 最大并发请求限制（默认 10）
- 使用 `p-limit` 库防止 API 过载
- 失败自动重试 3 次（指数退避：1s, 2s, 4s）

### 3.4 定时任务调度

**CronScheduler**：
- 处理心跳维持
- 支持周期性任务（如每日摘要、定期同步）
- 与 `cron` 工具集成实现自动化工作流

## 四、智能体层 (Agent Layer)

### 4.1 职责

系统的"大脑"，负责意图理解、任务拆解、记忆管理、执行循环。

### 4.2 ReAct 执行循环

```
┌─────────────┐
│  思考 (Think) │ ──→ 分析当前状态与目标
└─────────────┘
       ↓
┌─────────────┐
│  行动 (Act)  │ ──→ 选择并调用工具
└─────────────┘
       ↓
┌─────────────┐
│  观察 (Observe)│ ──→ 获取工具执行结果
└─────────────┘
       ↓
┌─────────────┐
│  反思 (Reflect)│ ──→ 更新策略与记忆
└─────────────┘
       ↓
    (循环或结束)
```

**Agent.run()** 控制整个流程，支持多轮迭代直至任务完成。

### 4.3 三级记忆系统

OpenClaw 最具工程特色的创新设计：

```
┌─────────────────────────────────────────────────────────────┐
│                    短期记忆 (Daily Log)                      │
│  每日日志文件 (2026-04-02.md)，追加式写入                    │
│  新会话自动加载"今天 + 昨天"，提供 48 小时连续感              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                    近端记忆 (Sessions)                       │
│  SQLite 存储完整会话存档，对话过长时关键信息被冲刷至此        │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                    长期记忆 (MEMORY.md)                      │
│  持久化知识库：技术栈偏好、项目决策、常用工具链              │
│  每次私聊时自动加载                                         │
└─────────────────────────────────────────────────────────────┘
```

**核心思想**：模型不需要知道所有事，只需要知道此刻最相关的事。

### 4.4 混合检索实现

**数据库位置**：`~/.openclaw/memory/{agentId}.sqlite`

**核心表结构**：

| 表名 | 说明 |
|------|------|
| `files` | 文件元数据（id, path, mtime, size, hash） |
| `chunks` | 文本块（id, file_id, start_line, end_line, text, hash, embedding） |
| `chunks_fts` | FTS5 全文搜索虚拟表 |
| `chunks_vec` | sqlite-vec 向量搜索虚拟表（embedding float[1536]） |

**检索流程**（`memory_search` 工具）：

1. **双路检索**：
   - BM25 关键词检索（FTS5）
   - 向量相似度检索（sqlite-vec，余弦相似度）

2. **结果合并**：取并集（而非交集），确保任一方法认为相关的内容均可进入候选池

3. **得分计算**：
   - BM25 归一化：`normalizedBm25 = 1 / (1 + bm25Score)`
   - 综合得分：`score = 0.7 * vecScore + 0.3 * normalizedBm25`
   - 向量权重 0.7，BM25 权重 0.3

4. **降级策略**：若 `sqlite-vec` 扩展未安装，回退至内存暴力余弦距离计算

## 五、执行层 (Execution Layer)

### 5.1 职责

调用具体技能（Skills）和工具（Tools），实现文件操作、代码执行、网络访问等实际能力。

### 5.2 工具系统 (Tools)

**总量**：26 个工具，通过 `tools.allow` 和 `tools.deny` 配置权限。

**Layer 1 - 核心能力 (8 个)**：

| 工具 | 功能 |
|------|------|
| `read` | 读取文件 |
| `write` | 写入文件 |
| `edit` | 编辑文件 |
| `apply_patch` | 应用补丁 |
| `exec` | 执行 shell 命令 |
| `process` | 进程管理 |
| `web_search` | 网络搜索 |
| `web_fetch` | 网页抓取 |

**Layer 2 - 高级能力 (18 个)**：

| 类别 | 工具 |
|------|------|
| 交互 | `browser`, `canvas`, `image` |
| 记忆 | `memory_search`, `memory_get` |
| 会话/Agent | `sessions_*`, `subagents`, `agents_list` |
| 通信/自动化 | `message`, `cron`, `gateway`, `tts` |
| 硬件/扩展 | `nodes`, `llm_task`, `lobster` |

### 5.3 技能系统 (Skills)

**定义**：技能是"教科书"，教导 OpenClaw 如何组合工具完成任务，**不直接赋予权限**（权限由 Tools 控制）。

**规模**：
- 53 个官方捆绑技能
- 13,700+ 第三方技能（ClawHub）

**加载机制**：
- 捆绑技能默认自动加载（若对应 CLI 工具已安装）
- 通过 `skills.allowBundled` 配置白名单模式

**安全建议**：ClawHub 自 2026 年 2 月起集成 VirusTotal 自动扫描，但建议安装前审查代码。

### 5.4 节点架构 (Nodes)

**本地节点**：
- 通过子进程或 Node 模块执行
- 直接访问本地资源

**远端节点**：
- 基于 WebSocket 长连接
- `NodeRegistry` 管理心跳与状态
- 支持跨设备控制（如 Raspberry Pi、Azure VM）

### 5.5 沙箱隔离机制

| 隔离层级 | 实现方式 |
|----------|----------|
| **权限隔离** | 工具决定实际能力，技能仅为指令手册 |
| **操作防御** | 高风险命令（如 `exec`）强制人工审批 |
| **部署隔离** | 建议部署在独立环境（如 Azure VM），与本地敏感数据物理隔离 |
| **授权管理** | 第三方服务通过 OAuth 授权，可随时撤销 |
| **容器隔离** | 默认 Docker 沙箱执行命令 |

## 六、插件化设计

### 6.1 Provider 插件系统

2026 年重构后的核心改进：

**标准接口**：
```typescript
interface LLMProvider {
  chat(messages: Message[]): AsyncIterator<Chunk>;
  supportTools(): boolean;
  initialize(config: ProviderConfig): Promise<void>;
}
```

**动态加载**：
- `ProviderLoader` 自动发现并注册 Provider
- 支持 Anthropic、OpenAI、Ollama、Kimi 等 22+ 模型
- 核心框架不再依赖具体模型 SDK

**效果**：核心包体积从 45MB 降至 8MB。

### 6.2 配置文件格式

| 配置类型 | 格式 | 位置 |
|----------|------|------|
| 权限配置 | JSON | `~/.clawdbot/exec-approvals.json` |
| 会话记录 | JSONL | `~/.clawdbot/sessions/*.jsonl` |
| 记忆文件 | Markdown | `~/.clawdbot/memory/MEMORY.md` |
| 技能配置 | YAML | `claw.yaml` |

## 七、核心运行链路

### 7.1 消息处理流程

```
用户消息 (Telegram)
    ↓
Channel 适配器 (标准化)
    ↓
Gateway (路由 + 入队)
    ↓
SessionManager (加载上下文)
    ↓
Agent.run() (ReAct 循环)
    ↓
Memory 检索 (相关上下文)
    ↓
Tool/Skill 执行
    ↓
结果返回 (原渠道)
```

### 7.2 自动化工作流示例

**定时任务触发**：
```
Cron (每日 9:00)
    ↓
触发器 → lobsters 工作流
    ↓
llm_task → 生成日报
    ↓
message → 发送到飞书群
```

## 总结

### 核心结论

1. **四层解耦架构**：交互层、网关层、智能体层、执行层职责清晰，支持独立演进与分布式扩展。

2. **三级记忆系统**：短期日志 + 近端会话 + 长期知识，结合 BM25+ 向量混合检索，实现高效本地化上下文管理。

3. **插件化优先**：Provider 插件系统支持 22+ 模型动态加载，核心包体积减少 82%。

4. **安全隔离**：工具权限与技能指令分离，高风险操作强制审批，Docker 沙箱默认隔离。

5. **TypeScript 全栈**：基于 Node.js 22+，采用 Express/Hono、BullMQ、SQLite 等成熟技术栈。
