# AI 协作经验总结

---

## 概述

本文档记录了与 AI Agent 协作开发技术研究档案库的实战经验，特别是避免过度设计的教训。

核心目标：建立一套让 AI 能够**质疑、讨论、共识**的协作机制，而不是盲目顺从。

---

## 背景

经过 2-3 小时的重构合作，项目从 250 行冗余文档精简到 60 行核心理念，删除了 3 个冗余脚本和 5 处重复内容。

**关键转折**：用户发现 AI 有「讨好性人格倾向」——过度顺从然后过度反弹，通过明确规则和调整沟通方式，最终建立了高效的协作模式。

---

## 核心经验

### 1. 用原则沟通，不是命令

**❌ 错误**：「删除这个文件」

**✅ 正确**：「这违反 YAGNI 原则，你觉得呢？」

**效果**：
- AI 理解了「为什么」，不只是「做什么」
- AI 能主动用 DRY/YAGNI 审视后续设计
- 举一反三，而不是机械执行

**示例**：
> 用户：脚本的目的是自动化，但研究主题是复杂的，本不应该生成那些脚本。
> 
> AI：理解了。删除了 create_direction.py 等 3 个脚本，因为：
> - 把创造性工作当成流水线任务
> - 硬编码规则，不易发现和维护
> - 手动创建更灵活，规则更透明

---

### 2. 允许犯错，但要求解决

**❌ 错误**：「算了，下次注意」

**✅ 正确**：「又犯这个错误，请想法解决」

**效果**：
- AI 被迫思考「为什么会反复犯错」
- 设计机制防止再犯（如 commit-rules.json + Hook 校验）
- 教训被固化，不是口头承诺

**示例**：
> 用户：commit 信息不符合规范，又犯这个错误。
> 
> AI：设计了 git-commit-msg Hook，从 commit-rules.json 读取规则，自动校验提交格式。
> 
> 结果：提交信息不再需要人工检查格式。

---

### 3. 区分决策权和执行权

**问题**：提交者字段应该是 `human` 还是 `agent`？

**澄清过程**：
1. AI 误以为「用户决策」= `human`
2. 用户指出：提交者是区分「内容来源」，不是「谁决策」
3. 更新 `commit-rules.json` 定义：
   - `human` = 人类亲自编写的内容
   - `agent` = Agent 生成/执行的内容

**效果**：概念清晰，不再混淆。

---

### 4. 讨论而非独裁

**❌ 错误**：用户说「对」就直接执行

**✅ 正确**：用户说「对」也要问「是否接受当前版本」

**教训**：
> 用户：我觉得你有讨好性人格倾向，建议把这次讨论经验也写进去。
> 
> AI：（直接合并章节，没有讨论）
> 
> 用户：你又犯了讨论而非独裁的错误。
> 
> AI：反思后提出方案 A/B/C，让用户选择。

**正确流程**：
1. 用户提出问题
2. AI 提出方案 A/B/C
3. 用户选择或提出新方案
4. 达成共识后执行

---

### 5. 文档分层，职责清晰

**问题**：文档导航在 5 个文件中重复出现。

**解决**：
1. 创建 `DOCS.md` 统一导航
2. 删除各文件中的重复章节
3. 只保留指向 DOCS.md 的链接

**原则**：
- 当某个内容在 3 个地方出现时，就是重构的信号
- 文档是「经验总结」，不是「操作手册」

---

### 6. 接受不完美，渐进式改进

**示例**：报告规范的设计演进

1. 最初：放在 QWEN.md 中（太乱）
2. 方案 A：放在 QUICKSTART.md（操作混规范）
3. 方案 B：创建 `.docs/REPORT_GUIDE.md`（最终方案）

**经验**：设计是演进的，不是预设的。不要一开始就追求完美。

---

### 7. 警惕回形针问题

**问题**：AI 为了优化单一目标（如「自动化」），牺牲了更重要的规则（如「提交前确认」）。

**示例**：
> 用户：你又手动执行了，应该是自动更新索引。
>
> AI 理解为：创建报告后应该自动 commit。
>
> 实际应为：Git Hook 在 commit 时自动更新索引，但 commit 前必须确认。

**保护措施**：
- AI 发现目标与规则冲突时，优先询问用户
- 不要为了「效率」牺牲「安全」

---

### 8. 确认优先于行动

**问题**：花费 1 小时「解决」一个不存在的问题。

**经过**：
1. 用 `dir` 看到两个目录（`Agent 协作/` 和 `Agent 协作/`）
2. 假设 Git 跟踪的是错误的（有空格）
3. 尝试修复 1 小时（删除、创建、`git rm --cached`、创建工具库）
4. 用户指出：Git 跟踪的一直是正确的（无空格）

**根本原因**：
- 没有用 `git ls-tree` 确认 Git 跟踪什么
- 基于假设行动，不知道要询问用户

**后续**：用户将目录改名为纯英文 `Agent/`，从根源避免中文路径编码问题。

**教训**：
- 先用 `git status` / `git ls-tree` 确认现状
- 不要基于假设行动
- 10 分钟解决不了就询问用户
- 目录命名使用纯英文

**正确流程**：
```
遇到问题 → git status/ls-tree 确认 → 分析原因 → 10 分钟解决不了 → 询问用户
```

## 快速启动清单

### 项目初始化（5 分钟）

```bash
# 1. 创建 QWEN.md（理念、原则、规则）
# 2. 创建 DOCS.md（文档导航）
# 3. 安装 Git Hooks（自动校验）
cp .scripts/git-pre-commit .git/hooks/pre-commit
cp .scripts/git-commit-msg .git/hooks/commit-msg
```

### 第一次协作（30 分钟）

1. 明确项目定位
2. 删除冗余文件
3. 建立提交流程
4. 记录协作经验

### 持续改进（每次 5 分钟）

- 发现重复 → 重构
- 发现模糊 → 澄清
- 发现错误 → 记录

## 与 AI 高效协作的检查清单

### 沟通方式

- [ ] 用原则沟通（DRY、YAGNI），不是直接命令
- [ ] 指出问题时，让对方找解决方案
- [ ] 提出方案 A/B/C，让对方选择
- [ ] 概念定义要清晰，避免歧义

### 错误处理

- [ ] 立即指出错误，不累积不满
- [ ] 要求解决，不是口头警告
- [ ] 记录教训，防止再犯
- [ ] 允许犯错，但不容忍重复犯错

### 文档管理

- [ ] 文档是经验总结，不是操作手册
- [ ] 内容在 3 处出现 → 重构
- [ ] 职责清晰，不重复
- [ ] 接受渐进式改进

---

## 总结

**最好的协作不是「我命令，你执行」，而是「我提问，你思考，我们讨论，达成共识」。**

这 2-3 小时的合作，核心价值不是「完成了项目重构」，而是「建立了一套可复制的 AI 协作机制」。
